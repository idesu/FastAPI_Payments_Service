# Асинхронный сервис процессинга платежей

Микросервис для асинхронной обработки платежей с гарантированной доставкой событий через Outbox pattern.

---

## Описание

Сервис принимает запросы на создание платежей через REST API, сохраняет их в PostgreSQL и публикует событие в RabbitMQ с гарантией доставки через **Outbox pattern**. Отдельный **Consumer** обрабатывает события: эмулирует обработку платежа (2-5 сек, 90% успеха), обновляет статус в БД и отправляет webhook‑уведомление на указанный URL с повторными попытками (3 раза, экспоненциальная задержка). При окончательной неудаче сообщение попадает в **Dead Letter Queue (DLQ)**.


Проект состоит из трёх независимых сервисов, каждый запускается в своём контейнере:

| Сервис | Назначение | Команда запуска |
|--------|------------|-----------------|
| **API** | FastAPI приложение, обработка REST запросов | `uvicorn app.main:app` |
| **Consumer** | Подписка на очередь `payments.new`, обработка платежей | `python -m app.consumer` |
| **Outbox Worker** | Фоновый процесс, публикующий события из таблицы `outbox` в RabbitMQ | `python -m app.outbox_worker` |
| **PostgreSQL** | Хранилище платежей и outbox записей | стандартный образ |
| **RabbitMQ** | Брокер сообщений с настройкой DLQ | стандартный образ |

**Взаимодействие:**
1. **API** получает запрос на создание платежа, сохраняет его со статусом `PENDING` и добавляет запись в таблицу `outbox` (в рамках одной транзакции).
2. **Outbox Worker** периодически (каждые 5 сек) забирает `pending` записи из `outbox`, публикует их в RabbitMQ и помечает как `sent`. Использует `SELECT ... FOR UPDATE SKIP LOCKED` для конкурентной обработки.
3. **Consumer** получает сообщение из очереди, эмулирует обработку (задержка, случайный статус), обновляет статус платежа в БД и отправляет webhook с ретраями. При неудаче – отправляет в DLQ.
4. **DLQ** (`payments.failed`) хранит сообщения, не обработанные после трёх попыток, для ручного анализа.

## Стек технологий

- **FastAPI** + **Pydantic v2** – REST API и валидация

- **PostgreSQL** – основная БД
- **RabbitMQ** – брокер сообщений
- **FastStream** – асинхронный клиент для RabbitMQ (consumer и publisher)
- **Alembic** – миграции БД

- **pytest** + **httpx** – тестирование

```bash

cd payment-service
cp .env.example .env
```


### 2. Запуск всех сервисов

```bash
docker-compose up --build
```

Будут подняты:
- PostgreSQL (порт 5432)
- RabbitMQ (порты 5672, 15672)
- API (порт 8000)
- Consumer

```bash
curl -X GET http://localhost:8000/docs
```

```bash
docker-compose down
```

## Архитектура

### Outbox pattern (гарантированная доставка)

1. При создании платежа запись сохраняется в БД, и в той же транзакции добавляется запись в таблицу `outbox` со статусом `PENDING`.
2. **Outbox Worker** каждые 5 секунд выполняет `SELECT ... FOR UPDATE SKIP LOCKED` для получения `PENDING` записей (не более 100 за раз).
3. Для каждой записи публикует событие в RabbitMQ (обменник `payments`, routing key `payment.created`).
4. При успешной публикации помечает запись как `SENT`. Если публикация упала – статус остаётся `PENDING`, и запись будет повторно обработана в следующем цикле.
5. Благодаря `SKIP LOCKED` несколько экземпляров воркера не дублируют обработку одной записи.

### Обработка платежа (Consumer)

1. Consumer подписан на очередь `payments.new` с ручным подтверждением (`ack_policy=AckPolicy.MANUAL`).
2. Получает сообщение, эмулирует обработку (случайная задержка 2–5 сек, 90% успеха).
3. Обновляет статус платежа в БД (`succeeded` или `failed`).
4. Отправляет webhook на указанный URL с повторными попытками (3 раза, экспоненциальная задержка).
5. Если webhook не удалось отправить после всех попыток – сообщение отправляется в DLQ (`payments.dead`).
6. При любых других ошибках (например, сбой БД) – выполняется ретрай с увеличением счётчика в заголовке `x-retry-count`. После 3-х попыток – в DLQ.

### Повторные попытки и DLQ

- **Webhook:** повторные попытки отправки уведомлений реализованы в сервисе `WebhookSender`).
- **Ошибки обработки:** логика повторных попыток реализована в consumer через ручное управление подтверждениями и заголовок `x-retry-count`.
- При превышении лимита (3 попытки) сообщение отклоняется (`reject(requeue=False)`) и, благодаря настройке `x-dead-letter-exchange` очереди, попадает в обменник `payments.dlx` и далее в очередь `payments.dead` (DLQ).

### Graceful shutdown

Каждый сервис корректно обрабатывает сигналы `SIGINT` и `SIGTERM`:
- **API:** через lifespan FastAPI (uvicorn) – закрывает соединения с БД.
- **Consumer:** через встроенный механизм FastStream (`app.run()`) – дожидается завершения текущей обработки.
- **Outbox Worker:** ручная обработка сигналов – завершает цикл после текущей итерации публикации.

## Особенности реализации

- **Модели SQLAlchemy** – используют `mapped_column` (SQLAlchemy 2.0 style) для улучшенной типизации.
- **Инверсия зависимостей** – сервисы принимают репозитории через конструктор, легко подменять для тестов и при изменении бизнес-требований.
- **Атомарность** – создание платежа и запись в outbox в одной транзакции; публикация и обновление статуса также атомарны.
- **Конкурентная обработка** – `FOR UPDATE SKIP LOCKED` предотвращает дублирование при нескольких экземплярах Outbox Worker.
- **Идемпотентность** – ключ `Idempotency-Key` гарантирует, что повторные запросы не создают новые платежи на платежном шлюзе.
- **Логирование** – настроено логирование всех ключевых действий.
